ENEMIES
Each enemy type uses the same core logic, and adds onto it depending on its specific behavior and characteristics.
MONSTER: Parent Controller

Each type of monster uses the same logic at its core. They all have access to functions that get triggered in the same way, overriding them as needed to make them behave in different ways from each other.

MONSTER: Illumination and Flashes

Each monster shares the same logic to check if they should react to the player's flashlight.

We start by checking if the active monster is in the middle of a jumpscare animation, to avoid overlapping actions that shouldn't happen at the same time.

Then, if the player's flashlight is currently active, we verify that the monster is within a specific cone in front of the player.

Depending on the results of that last check, we can update the Monster and tell it if the Player is currently illuminating it. Some monsters may use this to change their behavior (e.g. slowing down, freezing itself, getting stunned).

This check is also used to dictate if a monster can be impacted by a flash or not.

If the player FLASHES their flashlight while a monster is illuminated, it will run the 'OnFlash' function. Each monster inherits that function form the parent MonsterController class, and each are able to react in their own way.

MONSTER: Attacks and Jumpscares

The main ATTACK logic is also shared by every monster, and is usually triggered on contact with the player's hitbox.

We start by lowering the player's health. If the player's health goes below 0, we trigger a JUMPSCARE animation and kill the player.

If the player is still alive and the monster has an attack sound attached, we then play that sound to notify the player that they took damage.

If the monster is set to be destroyed on contact, we then Destroy the monster's root Game Object.

'HAND' MONSTER: Lifecycle

The behavior of the 'Hand' enemy is quite simple at its core, as most of the logic is handled by the parent MonsterController class.

A Hand monster starts by picking a spawn point randomly from a set list. Only a set amount of hands can be present at once in the level.

It then simply tries to make its way towards the player. Once close enough, it becomes a physics object and lunges towards the player.

When reaching the ground after a lunge, it'll be stunned for a few seconds, before restarting the cycle and going after the player.

After being FLASHED by the player or once the player leaves the area where these monsters reside, hand monsters will flee another possible spawn point picked at random. If the player hasnt left the area, they will then pick yet another spawn point and restart the entire cycle.



OTHER MONSTERS: Lifecycle

The other monsters are simple and/or similar enough to not require an entire section dedicated to them.

TRIPWIRES only damage the player on contact, and do absolutely nothing else.

THEO [INFECTED] only moves towards the player, and kills the player on contact. If stuck, he will also pick a random destination far away and try to go there for about 5 seconds, as a way to prevent him from camping doors.

THEO [INFECTED]: IK Spider-like legs

THEO's Spider-like legs use Unity's integrated RIG and IK Constraint systems.

The logic for it is simple, and is scalable for any amount of legs.

We only have to check the distance between the foot/tip and the first joint of each leg. If that distance is too long, we make that leg step forward. We get the exact stepping position using a Raycast.

While a foot is stepping forward, we also slightly arch its motion inwards, to make it look more like an actual step instead of just having the foot sliding forward.

Once a foot reaches its target destination, we play a stepping sound at that same position and repeat the cycle.
